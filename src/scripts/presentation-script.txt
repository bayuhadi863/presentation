SLIDE 2: PROBLEM STATEMENT (60 seconds)
================================================================================

[PAUSE - Allow time for audience to read the slide content.]

Let me begin by outlining the core challenges that motivated this research.

In modern software development, organizations increasingly face the challenge of 
deploying applications across multiple heterogeneous technology platforms. However, 
this multi-platform requirement creates three significant problems:

First, manual multi-platform development leads to inconsistency. When developers 
manually code for different platforms—such as ASP.NET Core backends and React TypeScript 
frontends—differences in implementation logic often emerge. This inconsistency causes 
what we call "logic drift," where the same business logic behaves differently across 
platforms, resulting in bugs and maintenance overhead.

Second, current development approaches are repetitively slow. Commercial low-code tools 
require extensive manual configuration for each platform variation, consuming valuable 
development time.

Third, we face fragmented ecosystems. Each platform typically operates within its own 
isolated tool and library ecosystem, making unified control and consistency extremely 
difficult to achieve.

[PAUSE - Let this sink in before advancing]

The core challenge we address is: "How can we maintain consistency across heterogeneous 
technology stacks while meeting short release cycles?"

SLIDE 5: CODEGEN WORKFLOW - MAIN SYSTEM FLOW (50 seconds)
================================================================================

[PAUSE - Reference the main sequence diagram.]

This slide presents the primary system workflow for code generation. Notice the 
message-driven architecture that enables asynchronous, scalable processing.

The flow is initiated when a user requests code generation through the frontend. This 
request is sent to the backend. Rather than generating code synchronously—which could 
cause timeouts for large requests—the backend publishes this request as a message to 
our message broker, specifically RabbitMQ.

[PAUSE]

The message broker ensures reliable, asynchronous message delivery. A dedicated Worker 
Service subscribes to these messages and processes the code generation tasks. Upon 
completion, the worker communicates with our Git Provider to create version control 
records.

[PAUSE]

This architecture provides several advantages: it decouples the frontend from long-running 
backend processes, enables horizontal scalability through multiple worker instances, and 
ensures reliable message delivery even during system failures.

[NEXT SLIDE]

SLIDE 6: LLM INTEGRATION & BATCHING OPTIMIZATION (50 seconds)
================================================================================

[PAUSE]

A key technical achievement in our system is the implementation of intelligent LLM 
batching strategy to optimize API performance and cost.

When our backend processes a code generation request, it first assembles context from 
two sources: the JSON metadata that defines what to generate, and the existing 
repository files that provide structural context. This context is then sent to the 
LLM API.

However, LLM APIs charge based on token usage, and large single requests become 
prohibitively expensive and risk exceeding token limits. Our solution implements a 
batching strategy: for backend code generation, we send the metadata in batches of 
five files at a time. This reduces token consumption per request while maintaining 
code consistency through context awareness.

[PAUSE]

The LLM returns JSON-formatted responses for each batch. Our backend then parses these 
JSON responses and assembles all generated files into a cohesive unit. This approach 
balances token efficiency, API cost optimization, and generation quality.

[NEXT SLIDE]

SLIDE 7: GIT PROVIDER OPERATIONS - AUTOMATED VERSION CONTROL (45 seconds)
================================================================================

[PAUSE - Reference the Git operations diagram.]

Our system automates the entire version control workflow. Once code generation is 
complete, the following automated Git operations execute:

First, the system parses the generated JSON to identify all files that need to be 
created or modified. Second, it creates a new Git branch with a unique identifier. 
Third, it builds comprehensive file actions—delete, create, or modify operations for 
each file.

[PAUSE]

Fourth, the system commits all changes atomically with a descriptive commit message. 
Fifth, it creates a pull request with a detailed description of changes, allowing 
developers to review before merging.

[PAUSE]

This automation eliminates manual Git operations and ensures consistent, traceable 
version control of all generated code.

SLIDE 8: JOB LIFECYCLE & STATUS TRACKING (45 seconds)
================================================================================

[PAUSE]

This slide illustrates the job lifecycle and how our system tracks code generation 
progress in real-time.

When a user initiates code generation, a job is created with an initial "waiting" 
status. This job record is stored in our database. Simultaneously, a message is 
published to RabbitMQ.

[PAUSE]

The worker service consumes this message from the queue, and the job status immediately 
transitions to "processing." During this phase, the worker performs actual code 
generation, batches requests to the LLM, receives responses, and prepares files.

[PAUSE]

Upon successful completion, the job status updates to "completed." The frontend can 
poll or subscribe to job status updates, providing users with real-time feedback about 
generation progress. If any error occurs, the status transitions to "failed" with error 
details logged.

This mechanism ensures users always have visibility into their code generation requests.
